[[chapter-ch-bank-link-fallback]]
=== Bank Link Fallback Behavior

Fallback behavior is activated when <<EVT-BANK-LINK-UNAVAILABLE,bank link is unavailable>> occurs during authorization. In this
condition Payment Engine must stop external authorization attempts and return a deterministic temporary
unavailable response to Checkout API.

Checkout API converts that response into user guidance that distinguishes transient infrastructure
failure from card or account decline. This prevents false negative messaging and reduces <<ACT-SUPPORT,support agent>> load
caused by misleading decline prompts.

This complements, but does not replace, the normal bank-link-up authorization path.

[[chapter-ch-decline-handling]]
=== Decline Handling And Support Flow

When <<EVT-PAYMENT-DECLINED,payment is declined>> is produced, Payment Engine returns a decline payload to Checkout API
containing <<ACT-CUSTOMER,customer>>-safe messaging and <<DATA-DECLINE-REASON,decline reason>> details. Checkout API then renders clear
feedback and retry guidance so users can recover without ambiguous failure states.

<<ACT-SUPPORT,support agent>> is linked to the same decline context to allow post-event triage, root-cause review, and
operational communication. This keeps user-facing behavior and <<ACT-SUPPORT,support agent>>-facing diagnosis aligned to the
same canonical event and data terms.

The chapter covers both <<ACT-CUSTOMER,customer>> messaging and <<ACT-SUPPORT,support agent>>-side operational follow-up.

[[chapter-ch-fraud-check]]
=== Fraud Check Integration

Fraud evaluation is a mandatory branch for transactions where <<FEAT-FRAUD-CHECK,fraud check is enabled>> is enabled. Payment
Engine calls Risk Scorer before final authorization, and the risk pipeline computes <<DATA-RISK-SCORE,risk score>>
from request metadata, amount, and behavioral signals.

If the resulting score indicates <<EVT-RISK-HIGH,risk score is high>>, the authorization path must transition away from
immediate approval and into controlled review handling. If the score remains below threshold, the
payment can continue through the normal bank authorization branch.

This chapter defines the boundary between fast-path processing and human-review escalation.

[[chapter-ch-payment-auth]]
=== Payment Authorization Flow

The payment authorization flow starts when a <<ACT-CUSTOMER,customer>> submits checkout data and the system raises
<<EVT-PAYMENT-AUTH-REQUESTED,payment authorization is requested>> while <<STATE-CHECKOUT-ACTIVE,checkout is active>> is true. Checkout API creates a payment
session key and forwards the authorization request to Payment Engine.

Payment Engine coordinates internal checks and sends the external authorization call to <<EXT-BANK-GW,bank gateway>>
only when <<STATE-BANK-LINK-UP,bank link is available>> is satisfied. The <<DATA-PAYMENT-ID,payment id>> term is carried end-to-end so every
service invocation and auditable decision can be correlated across logs, traces, and <<ACT-SUPPORT,support agent>> tooling.

This is the nominal happy-path before decline or review branches are considered.

[[chapter-ch-review-escalation]]
=== High-Risk Review Escalation

Review escalation applies when risk logic determines the transaction cannot be auto-approved. Payment
Engine places the transaction into <<STATE-REVIEW-PENDING,payment review is pending>> and emits <<EVT-SUPPORT-NOTIFIED,support agent is notified>> so manual
intervention is guaranteed and auditable.

<<ACT-SUPPORT,support agent>> receives enough context to decide whether to release, reject, or request additional
verification. This chapter intentionally describes a controlled pause state instead of a hard failure
so business workflows can continue safely under supervision.

Checkout API provides status visibility while <<ACT-SUPPORT,support agent>> decisions are made.

== Requirements Appendix


[[REQ-PAY-001]]
=== REQ-PAY-001

While <<STATE-CHECKOUT-ACTIVE,checkout is active>>, when <<EVT-PAYMENT-AUTH-REQUESTED,payment authorization is requested>>, the <<SYS-PAYMENTS,payments system>> shall create a payment session and request authorization from <<EXT-BANK-GW,bank gateway>>.


Notes: Core authorization path from checkout to <<EXT-BANK-GW,bank gateway>>.



[[REQ-PAY-002]]
=== REQ-PAY-002

Where <<FEAT-FRAUD-CHECK,fraud check is enabled>>, when <<EVT-PAYMENT-AUTH-REQUESTED,payment authorization is requested>>, the <<SYS-FRAUD,fraud detection system>> shall calculate a <<DATA-RISK-SCORE,risk score>> before approval.


Notes: Fraud scoring gate before approval.



[[REQ-PAY-003]]
=== REQ-PAY-003

If <<EVT-PAYMENT-DECLINED,payment is declined>>, then the <<SYS-PAYMENTS,payments system>> shall present a <<DATA-DECLINE-REASON,decline reason>> and a retry option to <<ACT-CUSTOMER,customer>>.


Notes: Decline handling and <<ACT-CUSTOMER,customer>> feedback.



[[REQ-PAY-004]]
=== REQ-PAY-004

While <<STATE-CHECKOUT-ACTIVE,checkout is active>>, when <<EVT-PAYMENT-AUTH-REQUESTED,payment authorization is requested>> and <<EVT-RISK-HIGH,risk score is high>>, the <<SYS-PAYMENTS,payments system>> shall place the payment in <<STATE-REVIEW-PENDING,payment review is pending>> state and notify <<ACT-SUPPORT,support agent>>.


Notes: Escalation flow for high-risk payments.



[[REQ-PAY-005]]
=== REQ-PAY-005

Where <<FEAT-FRAUD-CHECK,fraud check is enabled>>, when <<EVT-PAYMENT-AUTH-REQUESTED,payment authorization is requested>>, the <<SYS-PAYMENTS,payments system>> shall persist an audit record with <<DATA-PAYMENT-ID,payment id>> and <<DATA-RISK-SCORE,risk score>>.


Notes: Auditability for authorization decisions.



[[REQ-PAY-006]]
=== REQ-PAY-006

If <<EVT-BANK-LINK-UNAVAILABLE,bank link is unavailable>>, then the <<SYS-PAYMENTS,payments system>> shall present a temporary unavailable message to <<ACT-CUSTOMER,customer>>.


Notes: Fallback behavior when external banking path is down.
