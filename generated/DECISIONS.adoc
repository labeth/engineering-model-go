[[ADR-EMG-001]]
== ADR-EMG-001: Generate architecture decision records separately from the main architecture document

Status:: accepted
Date:: 2026-04-29

=== Context

Architecture decision records explain why durable model and generator choices were made.
The main architecture publication should stay focused on the current structure, views,
and evidence instead of embedding the full decision history inline.

=== Decision

Model architecture decision records as first-class authored data and generate them into
a separate Architecture Decision Records document. The main architecture document links
to the generated decision records document through a compact decision index.

=== Consequences

- The main architecture document remains focused on current architecture and traceability.
- Decision rationale remains reviewable as structured model content.
- Generation workflows need a separate decisions output alongside the main AsciiDoc output.

[[ADR-EMG-002]]
== ADR-EMG-002: Keep architecture decision records model-authored and separate from code links

Status:: accepted
Date:: 2026-04-29

=== Context

Architecture decision records capture durable reasoning about model and generator choices.
Linking decisions directly to implementation files would create noisy trace paths and imply
code-level ownership that decisions are not meant to carry.

=== Decision

Store architecture decision records as first-class authored model data with required context,
decision, status, date, and consequences. Do not add code-link fields to the ADR model.

=== Consequences

- ADRs remain focused on architectural rationale instead of implementation inventory.
- Code traceability continues to use requirement and owner markers.
- Main architecture outputs can link to ADRs without embedding code-level decision links.

[[ADR-EMG-003]]
== ADR-EMG-003: Render Gemara as a first-class output using the official go-gemara SDK types

Status:: accepted
Date:: 2026-06-27

=== Context

OpenSSF Gemara is a layered GRC engineering model with published CUE schemas and a Go SDK.
The engineering model already captures controls, threats, risks, attack vectors, functional
units, verification evidence, and POA&M items, which align with Gemara's L1-L3 catalogs, the L5
evaluation log, the L6 enforcement log, and the L7 audit log. We needed a way to emit Gemara that
is guaranteed structurally correct rather than a hand-written approximation.

=== Decision

Build Gemara documents from the official go-gemara SDK structs (github.com/gemaraproj/go-gemara)
and serialize them with the same YAML codec the SDK uses. Treat Gemara as a first-class rendering
with a library API, an enggemara CLI, an AsciiDoc chapter, and MCP tools. Validate every artifact
two ways: load it back through the SDK loader/type discriminator in tests, and run cue vet against
the published Gemara schemas.

=== Consequences

- Output is guaranteed to match the Gemara schemas and is consumable by other Gemara tooling.
- engmod takes a dependency on go-gemara (and transitively go-oscal) for the export path.
- The L5 evaluation log validates via the schema and SDK type discriminator but is not round-tripped through the SDK loader, because its AssessmentStep is a func type that serializes to a name string yet cannot be unmarshaled back.

[[ADR-EMG-004]]
== ADR-EMG-004: Source the Gemara evaluation log from authored control verifications, with pass and fail outcomes

Status:: accepted
Date:: 2026-06-27

=== Context

A Gemara L5 evaluation log is meant to carry the opinionated result of assessing each control,
including passing assessments. The existing OSCAL assessment results path intentionally records
only non-passing findings, so it is a lossy source for an evaluation log.

=== Decision

Build the evaluation log from authored control verifications, which carry both pass and fail
statuses, mapping each verification to an assessment log under its control's evaluation. Synthesize
a needs-review assessment for controls that have no authored verification.

=== Consequences

- The evaluation log faithfully represents pass and fail outcomes rather than failures only.
- Inferred-from-tests verification can later augment this source without changing the contract.

[[ADR-EMG-005]]
== ADR-EMG-005: Keep hand-written OSCAL SSP/AR/POA&M; add Gemara-to-OSCAL only as an additive bridge

Status:: accepted
Date:: 2026-06-27

=== Context

The go-gemara SDK converts a Gemara control catalog to an OSCAL Catalog and a Gemara evaluation log
to OSCAL Assessment Results. It does not produce an OSCAL System Security Plan or Plan of Action and
Milestones, and it does not carry system characteristics, POA&M items, or compliance-profile-resolved
reviewed controls with engmod's deterministic UUID scheme.

=== Decision

Retain engmod's hand-written OSCAL SSP, Assessment Results, and POA&M exporters unchanged. Expose the
Gemara-to-OSCAL conversion as an additive option on enggemara, emitting a Gemara-sourced OSCAL control
catalog (which engmod did not previously produce) and OSCAL assessment results.

=== Consequences

- No information is lost; the same compliance problems remain solved by the existing OSCAL outputs.
- A new Gemara-sourced OSCAL control catalog becomes available for catalog-centric workflows.
- There are now two OSCAL assessment-results paths; the engmod-native one stays authoritative.

[[ADR-EMG-006]]
== ADR-EMG-006: Synthesize Gemara structure that the model does not author, with explicit fallbacks

Status:: accepted
Date:: 2026-06-27

=== Context

Several Gemara constructs are required by the schemas but are not first-class in the engineering
model: per-control assessment requirements and applicability groups, threat capability references,
and risk severity, category, and appetite. Some of these are organizational inputs that cannot be
inferred from the technical model.

=== Decision

Synthesize the missing structure deterministically: assessment requirements from control
verifications (or a default requirement); applicability groups from functional units plus an
all-systems catch-all; threat capabilities from appliesTo with a synthetic system capability
fallback; risk severity derived from likelihood and impact; and a default risk category carrying an
authored appetite. Keep generated artifacts reproducible by stamping dates only from an explicit option.

=== Consequences

- Every artifact is schema-valid without requiring intrusive changes to the authored model.
- Risk appetite and applicability remain coarse until authored as first-class model inputs.
- A future model reshape can replace synthesized fields with authored ones without changing the export.

[[ADR-EMG-007]]
== ADR-EMG-007: Attach trace and model-link markers to package-level var and const declarations

Status:: accepted
Date:: 2026-06-27

=== Context

The Go code scanner originally treated only functions, methods, and types as declaration-level
trace targets. Markers placed before package-level var and const declarations (for example template
and pattern variables) did not attach and were reported as trace_unattached warnings, so their
linking intent was lost and the repository could not reach a fully clean strict-linking scan.

=== Decision

Treat package-level var and const declarations as declaration kinds that attach TRLC-LINKS and
ENGMODEL-LINKS markers, while keeping only functions and methods trace-required. Stray markers that
precede no declaration (for example a trailing marker at end of file) are still reported as unattached.

=== Consequences

- Existing var/const markers now link to the model, giving full linking and a warning-free scan.
- Functions and methods remain the only declarations that must carry a TRLC-LINKS marker.
- The scanner policy test was updated to detect stray markers via a non-declaration position.

[[ADR-EMG-008]]
== ADR-EMG-008: Model multi-repository work as a recursive system-of-systems

Status:: accepted
Date:: 2026-06-27

=== Context

DO-178C governs software only; the hardware and software platforms and their connections are
allocated at the system level (ARP4754A) above the software (DO-178C) and hardware (DO-254)
processes. The engineering model needs to represent system requirements, architecture, hardware,
and the hardware/software interface, and to compose software components that may live in
subdirectories or in separate repositories.

=== Decision

Treat every model uniformly as a system. A system may reference and allocate to subsystem models,
which are themselves systems, recursively to any depth. Software and hardware items are leaf systems
where the DO-178C and DO-254 lifecycle data attaches. There are no fixed system/software/hardware tiers.

=== Consequences

- One model shape composes at every level; existing single-model projects are simply leaf systems.
- Requirement flow-down across systems mirrors ARP4754A allocation.
- Composition is additive and backward compatible with existing models.

[[ADR-EMG-009]]
== ADR-EMG-009: Subsystem references are downward-only with parent-owned allocation

Status:: accepted
Date:: 2026-06-27

=== Context

A subsystem may be reused under many different parent systems and must not depend on any of them.
Allocation and binding must live where the integration happens.

=== Decision

A subsystem never references or is aware of its parents. A parent system declares which subsystems it
uses and how it allocates its requirements onto them. Subsystems expose a provides-and-requires
contract with a public surface distinct from internal identifiers; parents bind only to the public
contract and must satisfy each required interface.

=== Consequences

- Subsystem repositories are reusable and parent-agnostic.
- The reference graph is a parent-to-child directed acyclic graph; cycles are rejected.
- Validation has a standalone scope (the subsystem alone) and a composed scope (the bindings).

[[ADR-EMG-010]]
== ADR-EMG-010: Resolve subsystem references from local subdirectories first

Status:: accepted
Date:: 2026-06-27

=== Context

Composition should be proven within a single repository before adding cross-repository plumbing, and
certification evidence must be reproducible and resolvable offline.

=== Decision

The reference resolver supports local subdirectory paths and external git repositories. A git
subsystem is cloned once into the workspace .engmod cache (gitignored, treated as temporary), pinned to
an optional revision, and then resolved exactly like a local subsystem; an existing clone that points
at the same repository is reused without network access, so a build is offline-reproducible after the
first resolve.

=== Consequences

- The same resolver serves a monorepo and multi-repository layouts.
- Composed snapshots are reproducible; pin a revision for an exact one.
- Air-gapped certification environments remain supported once the .engmod cache is populated.

[[ADR-EMG-011]]
== ADR-EMG-011: Requirements carry no tiers; delegation is the link

Status:: accepted
Date:: 2026-06-27

=== Context

Requirement levels (system, high-level, low-level) are confusing across a system boundary: a parent
system's low-level requirement would point at a subsystem's system-level requirement, mixing tier
meanings between two models. The link itself already says whether a requirement is delegated.

=== Decision

Requirements carry no tiers. A requirement is either delegated to a subsystem through a downward
composition allocation or implemented here; the presence or absence of a delegation link is enough to
tell the two apart. Linking one requirement to another inside the same document is flagged, because it
recreates the tier coupling this decision removes.

=== Consequences

- Requirement documents stay flat and boundary-agnostic.
- Delegation is expressed once, by the integrating system's downward allocation.
- Internal requirement-to-requirement links are reported instead of silently accepted.

[[ADR-EMG-012]]
== ADR-EMG-012: Trace links are existence-checked, scoped to the owning model, and exported as a matrix

Status:: accepted
Date:: 2026-06-27

=== Context

Code trace markers (TRLC-LINKS to a requirement, ENGMODEL-LINKS to a model element) were only
shape-checked by a regex, so a link to a deleted or mistyped id passed silently. In a monorepo the code
root can also contain other models' code (each example carries its own architecture.yml), whose markers
reference their own requirements, not the model being generated.

=== Decision

Every code trace link is existence-checked against the model: a dangling requirement or model-element
link is an error, and a requirement traced by no code, verification, or delegation is reported as an
orphan warning. Code is attributed to the model whose architecture.yml is its nearest enclosing root,
so a parent model never validates or counts a child model's links. The consolidated trace is also
emitted as a machine-readable matrix (req to code, verification, and delegated subsystem requirement).

=== Consequences

- Trace links are trustworthy, not merely well-formed; the matrix is a real artifact, not prose.
- Monorepos with multiple models validate cleanly without cross-model false positives.
- Orphan coverage gaps are visible without breaking the build.

[[ADR-EMG-013]]
== ADR-EMG-013: CI enforces the generation gates and artifact freshness

Status:: accepted
Date:: 2026-06-28

=== Context

The tool's gates — engdoc errors, trace-link integrity, EARS lint, composition checks, and Gemara
schema validity — and its committed generated artifacts only have value if they are enforced. CI
previously ran gofmt, vet, and unit tests only, so a model edit that broke generation or left the
committed documents stale could merge green. Enforcing artifact freshness also requires generation to
be deterministic.

=== Decision

A single gauntlet (scripts/validate-all.sh) runs in CI and locally. It regenerates every model's
documents — engdoc must exit clean, which enforces trace integrity, dangling links, EARS, and
composition — exports each trace matrix with engtrace (no dangling links), fails on any drift between
regenerated and committed artifacts, and validates the Gemara and Structurizr exports against their
schemas when the tools are present. Inline term linking was made deterministic so the drift check is
stable.

=== Consequences

- The generation gates are load-bearing: a broken model or a stale committed document fails CI.
- Generated artifacts are deterministic and reproducible from the model.
- One script is the source of truth for local pre-commit checks and CI.
